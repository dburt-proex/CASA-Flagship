## 1. Introduction

CASA ("Cognitive Autonomous Safety Architecture") is an AI governance control plane designed to route agentic software through a three-gate decision system (Allow, Review, Halt). This Privacy Policy describes how CASA collects, processes, stores, and protects data in the course of its operation.

This policy applies to all operators, administrators, and integrated systems that interact with the CASA Control Plane.

---

## 2. Data We Collect

### 2.1 Operator Data
- **Authentication credentials:** Email addresses and role assignments used for login and role-based access control (RBAC).
- **Session tokens:** JSON Web Tokens (JWTs) issued at login containing role, email, and expiration metadata.
- **Interaction data:** Chat messages sent via the Operator Chat feature, policy simulation parameters, and decision replay queries.

### 2.2 System and Governance Data
- **Policy metadata:** Policy identifiers, simulation parameters, dry-run results, and impact analyses.
- **Decision audit records:** Immutable audit log entries capturing policy mutation events, including actor identity, IP address, timestamp, reason, and request correlation IDs.
- **Boundary stress metrics:** System health indicators, stress levels, critical boundary alerts, and recommendations.
- **Dashboard metrics:** Active policy counts, 24-hour decision volumes, boundary alert counts, and system status.

### 2.3 AI Interaction Data
- **Chat session history:** Conversations between operators and the AI assistant are stored for session continuity. Sessions expire after 1 hour.
- **AI-generated content:** Explanations, policy impact analyses, and approval briefs generated by the Google Gemini AI service.

### 2.4 Technical Data
- **Request correlation IDs:** Unique identifiers attached to every API request for traceability.
- **IP addresses:** Captured in audit log entries for critical write operations.
- **HTTP headers:** Standard request headers processed for routing and authentication.

## 4. Data Storage and Retention

### 4.1 Session Storage
- **Redis (production):** Chat session history is stored in Redis with a 1-hour TTL (time-to-live). Data is encrypted in transit via TLS (`rediss://` protocol).
- **In-memory fallback (development):** When Redis is unavailable, sessions are stored in application memory with a cap of 1,000 sessions to prevent memory exhaustion.

### 4.2 Audit Logs
- **Google Cloud Logging:** Critical policy mutation events are durably written to Google Cloud Logging under the `casa-audit-log` log name. These records are immutable and retained according to your Google Cloud retention policy (default: 30 days, configurable up to 3,650 days).
- **Fail-closed design:** If an audit log write fails, the associated action is rejected to maintain auditability.

### 4.3 Token Storage
- **Client-side:** JWT tokens are stored in the browser's `localStorage` under the keys `casa_token` and `casa_user`. Tokens expire after 7 days.
- **Server-side:** No long-term token storage. Tokens are verified on each request using cryptographic signature validation.

## 5. Third-Party Data Sharing

### 5.1 Google Gemini AI
- Chat messages and governance data are sent to the Google Gemini API for AI-assisted analysis when operators use the Chat, Explain, or Policy Analyze features.
- Data sent to Gemini is governed by [Google's Gemini API Terms of Service](https://ai.google.dev/terms).
- CASA does not send authentication credentials, tokens, or secrets to the Gemini API.

### 5.2 Python Backend Bridge
- Governance data (dashboard metrics, stress analysis, dry-run results, decision replays) is fetched from a Python FastAPI backend service.
- Communication with the backend includes request correlation IDs for traceability.
- No personal operator data is sent to the backend beyond request headers.

### 5.3 No Sale of Data
- CASA does not sell, rent, or trade any user data to third parties.

## 6. Data Security

### 6.1 Transport Security
- All external API communications use HTTPS/TLS.
- Redis connections in production use TLS (`rediss://` protocol).

### 6.2 Authentication Security
- JWT tokens are signed using HMAC-SHA256 with a configurable secret.
- Production deployments require a strong, unique `JWT_SECRET` (weak defaults are rejected at startup).
- Authentication endpoints are rate-limited (20 attempts per 15 minutes).

### 6.3 Application Security
- **Security headers:** Helmet.js enforces HTTP security headers (X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, etc.).
- **Rate limiting:** All API endpoints are rate-limited to 100 requests per minute per IP.
- **Input validation:** All inputs are validated using Zod schemas with enforced length constraints.
- **XSS prevention:** Markdown rendering strips raw HTML and blocks script, iframe, object, and embed elements.
- **Request body limits:** JSON request bodies are limited to 1 MB.
- **RBAC enforcement:** Admin write operations require admin role verification plus explicit confirmation code matching.

### 6.4 Sensitive Data Protection
- API keys are never logged or exposed via API endpoints.
- The debug environment endpoint has been removed to prevent accidental key exposure.
- Development login endpoints are disabled in production environments.

## 7. Your Rights

Depending on your jurisdiction, you may have the following rights:

- **Access:** Request a copy of the data CASA holds about you.
- **Rectification:** Request correction of inaccurate data.
- **Erasure:** Request deletion of your data (subject to audit log retention requirements).
- **Restriction:** Request limitation of data processing.
- **Portability:** Request your data in a machine-readable format.
- **Objection:** Object to specific data processing activities.

To exercise any of these rights, contact the system administrator responsible for your CASA deployment.

## 8. Data Breach Notification

In the event of a data breach that affects personal data:

1. Affected operators and administrators will be notified within 72 hours of discovery.
2. The notification will include the nature of the breach, data affected, and remediation steps.
3. The breach will be documented in the audit log for accountability.
